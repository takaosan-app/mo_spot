## 1. コンセプト

### アプリ名
**My Spot Our Spot**

### コアアイデア
「ちょっと時間があったときに撮って共有。他の人の発見を見てちょっと寄ってみる。」

写真を撮ってAIが何かを同定するアプリは多数存在するが、本アプリはさらに**分類して地図にプロットする**ことで、発見の共有と街・自然の探索を促す。

### ターゲット地域
**日本国内限定**。UI・通知・エラーメッセージはすべて日本語。App Store / Google Playも日本向けのみで申請。法的対応は日本法（個人情報保護法・消費者契約法）のみを対象とする。

### 2つのモード
| モード | 説明 |
|---|---|
| **My Spot** | 個人記録。非公開コレクション。自分だけの発見メモ帳。 |
| **Our Spot** | コミュニティ共有。他のユーザーの発見を地図で見て、「ちょっと行ってみよう」につながる。 |

この2モードはUIの軸であり、投稿単位で切り替えられる。

## 2. 対象・対象外

### 対象（投稿できるもの）
- 植物・花・樹木・菌類（この場所のサクラ、タンポポの群生など）
- 野鳥・昆虫・動物の目撃情報（今日ここでカワセミを見た）
- 建築・史跡・路上の発見（面白い意匠の建物、古い道標、変な看板）
- 自然現象（光の差し方、霧、霜柱）
- 路上アート・非公式だが公共的な場所（壁画など）
- 飲食店・商業施設（後述の条件付きで可）

### 飲食店・大型施設の扱い
**投稿は可能。ただし以下の設計で責任リスクを最小化する。**

Google Maps等で既に網羅されている施設情報との差別化のため、**実用的な構造化フィールドは持たない**。

| 設計方針 | 内容 |
|---|---|
| 構造化フィールドなし | 営業時間・料金・喫煙可否などの項目欄を設けない |
| 投稿の鮮度を常時表示 | 「○ヶ月前の投稿」をUI上で常に明示 |
| ユーザー発信の明示 | 「これは個人ユーザーの投稿です」をカード上に常時表示 |
| 公式感のあるバッジなし | 施設投稿に「公式情報」を示す見た目を与えない |
| 免責事項 | 登録時・投稿時に明示。施設情報の正確性はプラットフォームが保証しない旨を表示 |
| 通報・削除機能 | 明らかに誤った情報はコミュニティが報告できる |

> **法的留意点**: 日本では免責条項だけで全責任を回避できるわけではない（消費者契約法・景表法）。「実用フィールドを構造化データとして持たない」ことが最も有効なリスク低減策。コメント欄への自由記述は個人の表現として扱う。

## 3. AI機能

### 3-1. AIモデレーション（投稿前に実行・必須）

投稿フローの**最初のステップ**。モデレーションを通過した写真だけが分類・保存に進む。

| 判定対象 | 処理 |
|---|---|
| 顔・人物が写っている | 自動ブラー処理、またはアップロード拒否 |
| 車のナンバー・住所等 | 検出して投稿者に警告・修正を促す |
| 猥褻・暴力・汚物等 | 信頼スコアが閾値超で即却下 |
| 判定が曖昧（スコア中間値） | 投稿者に確認を求めるか、レビューキューへ |
| プライバシー侵害の疑い | 投稿者に通知・確認 |

**採用技術**: Google Cloud Vision API（SafeSearch + Face Detection）  
月1,000件まで無料枠。超過後は $1.50 / 1,000件。

### 3-2. AI自動分類（モデレーション通過後）

**全カテゴリを Claude Haiku 4.5（Vision）に統一する。**

| カテゴリ | 処理 | 補足 |
|---|---|---|
| 植物・花・樹木 | Claude Haiku で種名・科名・和名を生成 | 専用APIは商用コストが高く断念 |
| 野鳥・昆虫・動物 | Claude Haiku で種名・特徴・和名を生成 | iNaturalist CV APIは外部商用非公開 |
| 建築・史跡・路上の発見 | Claude Haiku で名称・様式・時代を生成 | |
| 自然現象・その他 | Claude Haiku で自由記述タグを生成 | |
| 飲食店・施設 | Claude Haiku で店舗名・ジャンルを推定 | Google Places参照のみ（書き込みなし） |

> **同定精度についての方針**: 本アプリは「汎用AIの範囲内での同定」と割り切る。植物・野鳥の専門的な同定精度を求めるユーザーはiNaturalist・PlantNet等の専門アプリを使うことを推奨。本アプリの価値は同定精度ではなく「発見の共有と地図」にある。

#### 和名取得フロー
```
Claude Haiku で学名・英名を取得
    ↓
iNaturalist API の ja 名（preferred_common_name）を優先参照
    ↓ 和名が空の場合
Claude Haiku にフォールバック（「この生物の日本語名は？」）
```

> **除外したAPI**: 生物多様性DB（GBIF Japan）はレスポンスが極めて悪いため対象外。

**分類信頼度スコア**をDBに保持。スコアが低いものは「同定中」バッジを表示し、コミュニティによる補足・訂正を促す。

### 3-3. コスト試算

## 4. 投稿フロー

```
写真を撮る／選ぶ
    ↓
① AIモデレーション（顔・有害・プライバシー判定）
    ↓ 通過
② AI自動分類（種・カテゴリ・タグを自動付与）
    ↓
③ 投稿者が確認・編集（タグ修正・コメント追加・公開設定選択）
    ↓
④ DB保存（GPS座標・分類・信頼スコア・タイムスタンプ）
    ↓
⑤ 地図にプロット（公開設定に応じて表示範囲が変わる）
```

## 5. 公開設定

投稿ごとに3段階から選択できる。

| 設定 | 地図表示 | 閲覧者 |
|---|---|---|
| **非公開** | 表示されない | 自分のみ（My Spotコレクション） |
| **フォロワーのみ** | フォロワーの地図に表示 | フォロー中ユーザー |
| **全体公開** | 全ユーザーの地図に表示 | 全員（Our Spot） |

### 位置情報の精度設定
希少種の生息地や個人宅付近の投稿プライバシーを守るため、場所の精度を選べる。

- **おおまかな場所**（街区レベル、約100m単位）
- **正確な場所**（GPS座標そのまま）

## 6. 地図・発見機能

### Our Spotマップ
- **カテゴリフィルター**: 植物のみ、鳥のみ、建築のみ、など
- **クラスタリング**: ズームアウト時はエリア単位でまとめる
- **季節フィルター**: 今の季節に見られるものだけ表示（例：3月の桜スポット）
- **スポット詳細**: 同じ場所に複数の投稿があればタイムライン表示

### 近くを探す
- 現在地から半径 500m / 1km / 5km
- 徒歩・自転車・車の所要時間目安
- 「今日の新着スポット」通知オプション

### タイムライン
- 新着順・人気順（いいね数）・フォロー中ユーザーの投稿順
- カテゴリ別フィード

## 8. ストレージ設計

### 採用サービス: Cloudflare R2

ドメインをCloudflareで管理しているため、同一アカウントでそのまま利用可能。初期費用・契約費・手数料は一切なし。S3互換APIのため実装コードはAWS S3とほぼ同一。

| 項目 | 内容 |
|---|---|
| サービス | Cloudflare R2 |
| 初期費用 | なし |
| 転送料（egress） | 永久無料・無制限 |
| ストレージ料金 | $0.015/GB/月（無料枠10GBまで無料） |
| 無料枠 | 10GB / 書き込み100万回 / 読み込み1,000万回（期限なし） |
| S3互換 | あり（既存S3ライブラリがそのまま使える） |

### 保存するファイルの種類と容量設計

| 種別 | 1枚あたり | 10万件 | 備考 |
|---|---|---|---|
| 投稿写真（圧縮後） | 約300〜500KB | 約30〜50GB | アップロード時に長辺1,024px以下に圧縮必須 |
| サムネイル | 約20KB | 約2GB | 一覧表示・地図ピン用 |
| **合計** | **約320〜520KB** | **約32〜52GB** | |

### スケール別月額費用（ストレージのみ・転送料は常に$0）

| 投稿数 | 保存量目安 | 月額費用 |
|---|---|---|
| 〜2万件 | 〜10GB | **無料枠内** |
| 10万件 | 約50GB | 約$0.60（約90円） |
| 100万件 | 約500GB | 約$7.50（約1,100円） |
| 1,000万件 | 約5TB | 約$75（約11,000円） |

### 画像圧縮の実装方針

アップロード時にサーバーサイドまたはクライアントサイドで以下を実施。これによりClaude Haikuへのinput tokenも約半減し、AI費用も同時に削減できる。

- 長辺を1,024px以下にリサイズ
- JPEG品質80%で再圧縮
- EXIFのGPS情報は「おおまかな場所」設定時に丸め処理してから保存

### 各レイヤーの選定理由

| レイヤー | 採用技術 | 理由 |
|---|---|---|
| アプリ | Flutter | iOS・Android・Web を単一コードベースでカバー。Google/Apple認証・カメラ・GPS・地図の実績あり |
| DB | Supabase（PostgreSQL + PostGIS） | PostGISで「近くを探す」地理空間クエリが得意。認証・リアルタイム・管理UIが標準搭載。PostgreSQL知識がそのまま活きる |
| 認証 | Supabase Auth（Firebase Auth も可） | Google/Apple ソーシャルログインに対応。Flutter SDKあり |
| 画像ストレージ | Cloudflare R2 | 転送料永久無料。ドメイン管理と同一アカウントで利用可能。S3互換 |
| AIモデレーション | Google Cloud Vision | SafeSearch + Face Detection。月1,000件まで無料 |
| AI分類 | Claude Haiku 4.5 Vision | 全カテゴリ統一。和名生成・説明文生成に対応。1投稿あたり約¥0.83 |
| 地図 | Google Maps Flutter plugin | 日本の地図・住所精度が最高。モバイルSDKは無制限無料。Flutterの公式pluginが安定 |
| プッシュ通知 | Firebase Cloud Messaging | Flutter対応の標準的な選択 |

## 12. 認証設計

### 採用方式: Google / Apple ソーシャルログイン

**方針**: メールアドレス・氏名・住所などの個人情報入力を不要にする。ワンタップでアカウント作成できるUXを実現する。

| 項目 | 内容 |
|---|---|
| 対応プラットフォーム | Google ログイン（Android・iOS両対応）、Apple ログイン（iOS必須） |
| アプリが受け取る情報 | 匿名ID（OAuth トークン）のみ。メールアドレスは非公開設定も可 |
| 表示名 | ユーザーが自由設定するニックネーム（個人情報に該当しない） |
| アバター | 任意（未設定可） |
| 実装ライブラリ候補 | Firebase Authentication、Auth0 |

**デバイスIDのみの運用を採用しない理由**: iOS の ATT（App Tracking Transparency）規制でデバイスIDによるユーザー識別がApp Store審査で問題になるリスクがある。機種変更・アプリ再インストール時にデータが引き継げない。グループ管理・BAN機能が機能しない。

### 取得する個人情報の範囲

本アプリが保持するのは以下のみ。氏名・住所・電話番号・生年月日は取得しない。

- OAuth プロバイダー発行の匿名ユーザーID
- メールアドレス（通知用・任意。非公開設定可）
- ユーザーが設定した表示名・アバター画像

## 15. 開発・クラウド契約の並行実行計画（早期費用発生を許容）

### 目的
- 開発実装とクラウド契約を同時進行し、後戻りコストを下げる
- 「無料枠で粘る」よりも、初期から小額課金で安定運用・検証速度を優先する

### 方針（意思決定ルール）
- 月次予算の上限を先に決める（例：**初期3か月は月2〜5万円**）
- 契約判断は「最安」より「開発速度・運用安定性・将来移行コスト」で行う
- 本番候補のサービスはMVP前から同一基盤で検証し、環境差異を減らす

### 実行フェーズ（開発と契約を揃える）

#### Phase A（0〜2週間）: 土台契約・検証環境を先に確保
- [ ] クラウド請求アカウント作成（個人/法人の支払い手段を確定）
- [ ] 予算アラート設定（50% / 80% / 100%）
- [ ] 監視・ログ基盤を有効化（障害原因の追跡を最優先）
- [ ] 開発/ステージング環境を分離

**契約対象（早期）**

- [ ] モデレーションAPI（Vision SafeSearch等）
- [ ] 認証基盤（Google/Appleログイン連携先）

**この時点で決めること**

### 対案（比較方針）

#### 対案A: 徹底的に無料枠中心で進める
- **利点**: 初期キャッシュ流出を最小化
- **欠点**: 本番想定の負荷/障害/請求挙動を検証しにくい、後半で設計変更が増えやすい

#### 対案B: 早期に小額課金を許容して本番近似で進める（本計画）
- **利点**: 実測ベースで判断できる、再設計リスクが下がる
- **欠点**: 早期に固定費/従量費が発生

**採用推奨**: 現状要望（開発と契約の足並みを揃える）に合致するため、**対案Bを推奨**。

### 技術マイルストーン優先の課金トリガー（Go / No-Go）

初期コストが低いサービス構成を前提に、課金判断は金額閾値より**技術的到達条件**を優先する。

#### Go判定（小額課金を開始してよい条件）
- [ ] **投稿パイプライン安定化**
  - `撮影/選択 → モデレーション → AI分類 → 保存 → 地図表示` のE2E成功率95%以上を7日連続で達成
- [ ] **運用可観測性の確立**
  - API失敗率・投稿成功率・1投稿原価を常時計測（ダッシュボード化）
  - 予算アラート（50% / 80% / 100%）が実際に通知されることを確認
- [ ] **劣化運転の実装完了**
  - API障害または予算閾値到達時に、段階制限（例：再試行キュー化、補助生成の簡略化）が自動で有効化される

#### No-Go判定（課金拡大を止める条件）
- [ ] API失敗率が3%を超過した状態が24時間以上継続
- [ ] 投稿成功率が90%未満に低下
- [ ] 1投稿原価が想定値を20%以上上振れし、原因未特定

#### 運用ルール（シンプル版）
- **課金開始**: Go条件3項目をすべて満たした時点
- **課金拡大**: 日次投稿数が想定の70%を7日連続で超過
- **課金抑制**: No-Go条件のいずれかに該当した時点で段階制限を適用
